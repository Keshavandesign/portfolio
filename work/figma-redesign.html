Mini Project Figma Collaboration Systems

Figma comments
have no lifecycle.

You open a file for handoff and find 47 unresolved comments. Some are from three weeks ago. Some are already fixed. Some are still blocking. There's no way to tell which is which without reading every single thread. Design reviews in Figma never formally close.

Type

Mini Project · 2 days

Problem

Comment graveyard at handoff

Root cause

Comments have no state machine

The real problem

A comment is either
open or resolved. That's not enough.

Figma's current comment model is binary: resolved or not. But real design reviews aren't binary. A comment can be acknowledged but not actioned, blocking the handoff, a question that was answered verbally, or feedback that was intentionally rejected.

Without states for these situations, every comment looks the same. Developers inherit a comment list with no signal. Designers re-read old threads to reconstruct decisions they already made. The file becomes a comment graveyard.

01
Reviews never close
There's no concept of a "review session" in Figma. Comments accumulate across multiple review rounds with no clear boundary between them. The file carries all of them forever.
02
Resolved means nothing
Resolving a comment hides it. Designers use it to declutter, not to signal intent. "Resolved" could mean fixed, rejected, or just dismissed. Developers can't trust it.
03
Handoff has no checklist
When the file goes to engineering, there's no summary of what was decided, what changed, and what the developer needs to know. They have to audit the whole file themselves.

"I spent 40 minutes before handoff just reading old comment threads trying to figure out which ones were still relevant. That's not design work."

Synthesised from observations with 6 product designers across 3 companies

The redesign

Give comments a
state machine.

Four states replace the binary open/resolved model. The transition between states is the signal: a comment moves from Open to Blocking when it must be fixed before handoff, to Accepted when a decision is logged, or to Resolved when it's done. At handoff, the Blocking list auto-generates a developer checklist.

I considered a simpler "priority flag" approach: just marking some comments as urgent. The problem: urgency without lifecycle still doesn't tell developers which comments are still relevant. The state machine costs one extra mental model but solves the actual graveyard problem that flags don't.

Open
New feedback, needs attention. Default state for all comments.
→
Blocking
Must be resolved before handoff. Escalated by designer or reviewer.
→
Accepted
Decision logged. Feedback acknowledged, change confirmed or declined with reason.
→
Resolved
Done. The change is in the file. Safe to archive.

The Problem

47 open comments.
No way to read them.

Every comment pin looks the same. Purple dot. No state. You hover to read the text, hoping for context. The sidebar shows a flat list sorted by position, not urgency. There's no filter for "what's blocking me right now."

Developers either ignore the comment panel entirely (building from what they see) or ask the designer to walk them through it. Neither scales.

The Redesign

Review Mode.
States that do work.

Comments carry state. Pins change color: red for Blocking, purple for Open, faded blue for Resolved. The sidebar groups by state instead of position.

When you mark a file ready for handoff, Figma auto-generates a checklist from every Blocking comment that was Accepted, giving developers a structured summary of what changed and why, without anyone writing docs by hand.

Honest limits

Where this
breaks down.

A design solution that doesn't know its own failure conditions isn't fully thought through. These are the scenarios where this redesign stops working, and what a V2 would address.

Limit

Solo designers and quick-turnaround files

For a 2-hour sprint with one designer and no formal review process, four comment states is overkill. Review Mode should auto-collapse for files with fewer than 10 comments. I didn't solve this boundary condition.

Limit

Reviewer adoption

The checklist auto-generation only works if designers consistently mark comments as Blocking before closing them. If that habit doesn't form (and new habits rarely form without reinforcement), the checklist never appears. The design depends on a behavior change I haven't solved for.

Limit

Teams that don't use comments at all

Some engineering teams don't look at Figma comments. They treat them as the designer's private scratchpad. Adding states to those comments doesn't fix the graveyard; it just adds structure to noise. The root cause there is process, not tooling.
